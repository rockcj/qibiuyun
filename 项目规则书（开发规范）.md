## 一、项目概述与核心原则

### 1.1 项目定位

OfferGPT 是一款基于 **AI 实时语音交互** 的英语口语陪练工具，专注于 **真实对话场景** 下的沉浸式训练。核心差异化：场景可选 + 简历/JD 驱动（面试模式）+ 实时语音 + 多维纠错 + 可量化复盘。

### 1.2 核心设计原则

| 原则 | 说明 |
|------|------|
| **低延迟优先** | 实时语音链路必须流水线并行，ASR→LLM→TTS 不得串行阻塞 |
| **主链路轻量** | 实时问答只带必要上下文，分析类 Agent 异步执行不阻塞 TTS |
| **降级兜底** | 每个外部依赖（ASR/LLM/TTS）必须有降级路径，Demo 现场不能崩 |
| **可解释** | 评分必须绑定证据（transcript snippet + 音频片段 + VAR 事件） |
| **配置驱动** | Persona、场景、评分配置不可硬编码，须通过配置文件或数据库驱动 |
| **独立完成** | 代码、功能必须自主实现，引用第三方须在 README 和 PR 中注明来源 |

### 1.3 语言规范

| 范围 | 语言要求 |
|------|----------|
| 代码注释 | **必须** 使用中文 |
| PR 标题、描述、提交信息 | **必须** 优先使用中文 |
| 前端错误提示 | **必须** 使用中文 |
| API 错误响应 `message` 字段 | **必须** 返回中文 |
| 启动配置缺失等致命错误 | **必须** 输出中文错误信息 |
| 用户界面 | 支持中文/英文切换 |

## 二、核心用户流程

```
用户选择场景（面试 / 点餐 / 会议）
  ↓
（若为面试场景）上传简历 + 输入 JD → 系统解析
（若为点餐/会议场景）选择角色/主题
  ↓
系统生成个性化对话计划（问题池 / 对话大纲）
  ↓
用户选择 AI 人格 / 角色风格
  ↓
进入实时英语对话
  ↓
AI 根据用户回答动态追问 / 引导
  ↓
系统记录语音、文本、时间戳与行为数据
  ↓
生成课后报告（发音/语法/结构/流畅度等）
  ↓
查看时间轴回放（Interview VAR）
  ↓
获得成长路线图
```

> 来源：PRD v2.0 第三章

### 2.1 场景支持矩阵

| 场景 | scene ID | 触发简历/JD | 角色示例 | P0 要求 |
|------|----------|-------------|----------|---------|
| 求职面试 | `interview` | 是 | Founder, Product Thinker, Engineering Leader, Stress | ✅ 完整可用 |
| 餐厅点餐 | `restaurant` | 否 | 友好/忙碌/不耐烦服务员 | ✅ 完整可用（P1） |
| 商务会议 | `meeting` | 否 | 主持人、同事、上级 | ✅ 完整可用（P1） |

> 三场景均有独立 Prompt 追问策略（面试 5 条 / 点餐 6 条 / 会议 6 条）、完整 Topic→Goal 映射（全覆盖 18 个 topic）、预置 Demo 数据（3 个 seed 会话）。

## 三、开发流程规范

### 3.1 开发阶段优先级

```
P0（面试场景闭环）→ P1（多场景+差异化）→ P2（冠军功能）
```

**铁律：P1/P2 的新增功能不得导致 P0 面试场景的接口、实时链路、报告不可用。**

### 3.2 各阶段准入条件

#### P0 阶段 — 面试场景闭环（必须全部满足）

- [ ] 首页可看到面试场景入口并进入配置流程
- [ ] 支持上传简历（PDF/TXT）、输入 JD、选择 Persona、创建会话
- [ ] `GET /api/scenes` 至少返回可用的 `interview` 场景配置
- [ ] `POST /api/interviews` 可创建面试会话
- [ ] WebSocket 可接收 `audio.input` 并返回字幕或 AI 文本
- [ ] 实时面试页展示用户字幕、AI 回复、轻纠正提示
- [ ] 结束会话后生成 Offer Report 和 VAR 事件
- [ ] 面试报告页展示 Offer 评分、维度分数、建议、VAR 时间轴
- [ ] 关键 Agent 调用写入 `agent_logs` 表
- [ ] `restaurant` / `meeting` 已完整实现，不影响面试闭环

#### P1 阶段 — 多场景差异化

- [x] 餐厅点餐和商务会议场景均可正常进入并完成 ≥10 轮对话，各有独立追问策略
- [x] 不同场景使用各自的评分维度和权重（面试 6 维 / 点餐 5 维 / 会议 5 维）
- [x] VAR 时间轴可按颜色区分事件类型、点击定位
- [x] 支持多 ASR 模型运行时切换
- [x] 三场景 Demo 数据（各含完整对话 + 报告 + VAR 事件）
- [x] Demo 页面支持三场景切换器
- [x] 三场景 topic→goal 映射全覆盖（18 个）
- [x] 场景配置页移除占位标签，新增氛围描述卡片

#### P2 阶段 — 冠军功能

- [ ] Interview Twin 数字分身
- [ ] Growth Coach 成长路线图
- [ ] 压力面试模式（真实打断控制）

### 3.3 每日开发节奏（72h 参考）

| 时间段 | 目标 | 产出 |
|--------|------|------|
| 0-12h | 搭建骨架和场景数据模型 | Next.js + FastAPI + PostgreSQL + Scene 配置 + 基础 API |
| 12-24h | 场景选择与面试资料链路 | 三场景入口 + 简历/JD 解析 + Persona 配置 + 创建会话 |
| 24-36h | 实时语音链路 | WebSocket + ASR + Scene Router + LLM + TTS |
| 36-48h | Agent 与实时轻纠正 | Conversation Agent + Interview Agent + Grammar 策略 |
| 48-60h | 报告和 VAR | Timeline Events + 场景评分 + Report UI |
| 60-68h | 多场景打磨 + Demo 稳定 | 餐厅/会议 Prompt + 预置数据 + 备用路径 + 异常处理 |
| 68-72h | 彩排与性能检查 | 演示脚本 + 监控面板 + 答辩材料 |

## 四、代码风格规范

### 4.1 命名规范

| 对象 | 规范 | 示例 |
|------|------|------|
| 变量名、方法名 | **驼峰命名法 (camelCase)**，必须具备描述性 | `getUserProfile()`, `sessionToken` |
| JSON 字段名 | **驼峰命名法 (camelCase)** | `sessionId`, `parseStatus` |
| API 路径 | **小写复数名词** | `/api/scenes`, `/api/interviews` |
| 数据库表名 | **小写蛇形命名 (snake_case)** | `timeline_events`, `agent_logs` |
| 数据库列名 | **小写蛇形命名 (snake_case)** | `user_id`, `created_at` |
| 类名、组件名 | **大驼峰 (PascalCase)** | `ConversationService`, `VoiceSessionPanel` |
| 文件名 | **小写蛇形命名 (snake_case)** Python / **大驼峰 (PascalCase)** React 组件 | `asr_service.py` / `SceneCard.tsx` |

### 4.2 ID 命名约定

| 上下文 | 使用名称 | 说明 |
|--------|----------|------|
| 对外接口（REST/WS） | `sessionId` | 前端、API 响应、WS 消息统一使用 |
| 数据库内部 | `interview_id` / `id` | 内部列名，接口层须做转换 |
| 前端变量 | `sessionId` | 与后端 DTO 字段保持一致 |

### 4.3 注释规范

| 要求 | 说明 |
|------|------|
| **必须** 使用中文注释 | 所有注释均为中文 |
| **必须** 为关键逻辑添加注释 | 复杂判断、异步流程、业务规则必须有中文注释 |
| **禁止** 为显而易见的代码添加注释 | 如 `# 给 x 赋值 1` 这种无意义注释 |
| **必须** 保持注释与代码一致 | PR 修改代码时同步更新注释 |

### 4.4 枚举值规范

| 枚举 | 允许值 |
|------|--------|
| `scene` | `"interview"` \| `"restaurant"` \| `"meeting"` |
| `sessionStatus` | `"created"` \| `"connecting"` \| `"running"` \| `"finishing"` \| `"completed"` \| `"failed"` |
| `parseStatus` | `"pending"` \| `"success"` \| `"failed"` |
| `difficultyLevel` | `"junior"` \| `"middle"` \| `"senior"` |
| `eventType` | `"pronunciation"` \| `"grammar"` \| `"filler_word"` \| `"highlight"` \| `"llm_detected"` |

## 六、PR 规范

### 6.1 PR 核心规则

| 规则 | 说明 |
|------|------|
| **一个 PR 只做一件事** | 只实现或修改一个独立功能，禁止混入多个无关功能 |
| **大功能必须拆分** | 拆为多个小 PR，按可运行、可验证的阶段逐步提交 |
| **合并后 main 必须可运行** | 评委在任意时间查看都应能复现演示效果 |
| **必须基于 PR 添加功能** | 禁止直接把大批量代码一次性导入 main |

### 6.2 PR 描述必须包含

1. **标题**：一句话说明本 PR 新增或修改了什么
2. **功能描述**：说明该功能的作用、入口和使用方式
3. **实现思路**：简要说明技术选型、核心逻辑或关键代码结构
4. **测试方式**：说明如何运行、如何验证功能正常

### 6.3 禁止事项

- ❌ PR 描述空白
- ❌ PR 描述与实际代码变更严重不符
- ❌ 最后一天一次性导入全部代码
- ❌ 未说明依赖来源、第三方框架或复用代码来源
- ❌ 一个 PR 同时混入多个无关功能、格式化、重构或文档变更
- ❌ 提交密钥、账号、`.env`、构建缓存或无关临时文件

### 6.4 提交前检查清单

合并或创建 PR 前**必须**确认：

- [ ] 项目可以正常安装依赖并运行
- [ ] 本 PR 的功能可以单独验证
- [ ] README 已列明依赖、运行方式和原创功能说明
- [ ] PR 描述完整，且和代码变更一致
- [ ] 没有提交密钥、账号、`.env`、构建缓存或无关临时文件
- [ ] P0 阶段：已完成从首页到面试报告的主流程验证
- [ ] P1/P2 阶段：额外完成面试场景回归验证
- [ ] 如引入第三方依赖，README 中已列明依赖来源和原创边界

## 七、架构约束

### 7.1 前后端职责边界

| 层 | 负责 | 禁止 |
|----|------|------|
| **前端** | 用户交互、状态展示、浏览器端音频采集 | **禁止** 在浏览器中实现核心评分、Agent 推理、报告生成 |
| **后端** | 业务逻辑、实时链路、Agent 编排、数据持久化 | **禁止** 处理页面布局和视觉展示 |

### 7.2 前端状态管理

- 创建会话后，前端状态**必须**围绕 `sessionId` 管理
- 页面不再依赖本地表单状态，转而使用服务端返回的 `sessionToken`、`websocketUrl`、`sceneContext`

### 7.3 Agent 同步/异步规则

| Agent | 执行时机 | 同步/异步 |
|-------|----------|-----------|
| Scene Router Agent | 创建会话时 | **同步** |
| Persona Agent | 创建会话时 | **同步** |
| Conversation Agent | 每轮用户回复后 | **同步**（实时链路） |
| Interview Agent | 面试追问时 | **同步**（实时链路） |
| Grammar Agent | 每轮回复后 | **异步**（严重错误可回传轻纠正） |
| Pronunciation Agent | 每轮回复后 | **异步** |
| STAR Agent | 行为问题回复后 | **异步** |
| Replay Agent | 收到分析信号后 | **异步** |
| Report Agent | 会话结束后 | **异步或同步等待** |
| Growth Coach Agent | 报告生成后 | **异步** |

**铁律：实时主链路只允许低延迟 Agent 同步参与；分析类 Agent 必须异步执行，不得阻塞 TTS 回复。**

### 7.4 实时语音链路约束

- 音频格式：16kHz 单声道 PCM16
- VAD：客户端 + 服务端双层能量检测
- Turn 边界判断：静音 ≥500ms 或客户端主动发送 `audio.turn.end`
- ASR 识别：后台线程执行，不阻塞主事件循环
- LLM 上下文：实时链路只带最近 5-6 轮对话 + 滚动摘要
- TTS 分句播放：按标点切分，首句生成即下发

### 7.5 降级路径（必须实现）

| 故障场景 | 降级方案 | 用户感知 |
|----------|----------|----------|
| ASR 不可用 | 前端切为文本输入 | "语音识别暂不可用，已切换为文本输入模式" |
| TTS 不可用 | 返回 `agent.text.delta` + 浏览器 SpeechSynthesis | "已切换为浏览器内置语音朗读" |
| LLM 超时 | 使用场景问题池生成保守追问 | 回复略保守但不中断对话 |
| 无麦克风权限 | 自动显示文本输入框 | "麦克风权限被拒绝，请检查浏览器设置" |
| WebSocket 断开 | 自动重连 3 次（间隔 3s），恢复会话状态 | "连接断开，正在重连…（第 N/3 次）" |
| 报告生成失败 | 规则引擎兜底评分 | 后端自动处理，用户无感知 |
| 后端不可用 | `/demo` 页面使用 localStorage 缓存 | "Demo 演示数据 — 离线缓存模式" |
| Redis 不可用 | 自动降级为内存缓存 | 后端自动处理，用户无感知 |

### 7.6 Demo 现场稳定性要求

- [ ] 无麦克风时 → 给出明确中文提示 + 允许切文本输入或预置 Demo
- [ ] WebSocket 断开时 → 自动重连一次 + 显示恢复状态
- [ ] TTS 播放失败时 → 仍显示 AI 文字回复
- [ ] 报告未生成时 → 报告页显示加载状态 + 轮询 API
- [ ] 所有错误提示**必须**使用中文

## 八、API 契约规范

### 8.1 通用约定

| 约定 | 说明 |
|------|------|
| 数据格式 | REST 请求和响应默认 JSON |
| 时间格式 | ISO 8601 字符串或毫秒时间戳 |
| 鉴权 | Bearer JWT Token（生产）；Demo 模式可跳过 |
| 错误响应 | **必须**包含 `errorCode`、`message`（中文）、`requestId` |
| WS 消息 | **必须**包含 `type` 和 `sessionId` |

### 8.2 字段命名

- JSON 字段**必须**统一使用 camelCase
- API 路径**必须**使用小写复数名词
- 前端变量、后端 DTO 字段、文档示例**必须**保持相同名称
- 对外统一使用 `sessionId`；数据库内部使用 `interview_id` 须在接口层转换

### 8.3 API 变更规则

- 每次 API 变更**必须**同步更新 `docs/agent-team/api-contract.md`
- 后端实现**必须**保持响应字段与文档示例兼容
- 前端**必须**能仅凭 API 契约文档完成 Mock 数据开发
- 任何失败响应**必须**返回中文 `message`

### 8.4 场景接口特殊规则

- P0 阶段**必须**优先保证 `interview` 接口可用
- `restaurant` 和 `meeting` 未启用时**必须**返回明确的禁用状态
- 场景枚举值：`"interview"` | `"restaurant"` | `"meeting"`

## 九、配置与环境管理

### 9.1 配置加载策略

```
代码默认值 → 环境变量覆盖 → 启动时校验 P0 必需项
```

- 代码中只保留非敏感的默认值（默认场景、默认模型名、超时值、功能开关）
- 所有服务地址、账号密码、API Key、TOS Key 等敏感信息**必须**通过环境变量注入
- 敏感信息**禁止**写入代码、README 或 PR 描述

### 9.2 环境文件规则

| 文件 | 规则 |
|------|------|
| `.env.local` | **禁止**提交到 Git（已在 `.gitignore` 中） |
| `.env.secrets.local` | **禁止**提交到 Git，用于队友间分享凭据 |
| `.env.example` | **禁止**包含真实密钥、密码或 Token |
| `.env.docker` | 可提交，仅含占位符 |

### 9.3 启动校验

启动时**必须**校验 P0 必需的环境变量，缺失时输出中文错误并拒绝启动。

Demo 模式下允许 ASR/TTS/LLM 使用 mock 回退，但数据库和缓存的配置仍需提供。

### 9.4 关键配置检查项

- [ ] `DATABASE_URL` 指向正确的 PostgreSQL 实例
- [ ] `REDIS_URL` 以 `/1` 结束（数据库编号）
- [ ] `LLM_API_BASE_URL` 为 `https://api.deepseek.com/anthropic`
- [ ] `TOS_ACL` 保持 `private`（除非明确需要公开静态资源）
- [ ] `CORS_ALLOW_ORIGINS` 生产环境限制为实际域名
- [ ] `APP_ENV=production` 时 mock 开关全部为 `false`

### 9.5 Demo 降级开关

| 开关 | 用途 |
|------|------|
| `ENABLE_TEXT_INPUT_FALLBACK` | ASR 不可用时允许文本输入 |
| `ENABLE_MOCK_ASR` | Demo 模式使用 mock ASR |
