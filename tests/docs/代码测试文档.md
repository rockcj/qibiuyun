# OfferGPT 代码测试文档

## 一、测试目标

本文档用于评估 OfferGPT Step 2「场景配置与简历/JD 解析（面试优先）」的前后端代码质量、功能完整性、接口契约一致性、模块前后逻辑严谨性、潜在冲突与风险，并给出后续优化建议。

本阶段测试优先覆盖 P0 面试场景闭环：

1. 场景配置加载与选择。
2. 简历上传、文本抽取和结构化画像解析。
3. JD 输入、岗位画像解析和难度推断。
4. 面试会话创建，绑定 `resumeId`、`jobId`、`personaMode`。
5. 点餐/会议占位场景可创建轻量会话。
6. 所有错误响应符合 `errorCode`、中文 `message`、`requestId` 契约。

## 四、前后逻辑严谨性与冲突检查

### 1. 前端到后端数据流

面试场景数据流为：

```text
/scenes/interview
  → SceneConfigForm 收集 topic、roleMode、difficultyLevel
  → /interview/setup 上传简历并解析 JD
  → POST /api/interviews 创建会话
  → /sessions/{sessionId} 展示会话占位信息
```

该链路中前端传参字段与 `api-contract.md` 使用同一套 camelCase 命名，后端 DTO 也使用 `scene/topic/roleMode/resumeId/jobId/personaMode`，不存在命名冲突。

### 2. 后端数据库关系

`Interview.resume_id` 和 `Interview.job_id` 指向 `Resume`、`Job` 表。面试创建时会先校验 UUID 格式，再校验记录是否存在，可避免脏引用。

### 3. P0 与 P1/P2 边界

`interview` 必须绑定简历/JD；`restaurant`、`meeting` 不绑定简历/JD。当前代码用 `req.scene == "interview"` 做强校验，避免 P1/P2 占位逻辑影响 P0 面试主链路。

### 4. 错误格式一致性

后端通过 `ApiError`、HTTP 异常处理器和参数校验处理器统一响应：

```json
{
  "errorCode": "RESUME_REQUIRED",
  "message": "面试场景需要提供简历 resumeId",
  "requestId": "req_xxx"
}
```

前端 API 客户端优先读取 `message`，可保证用户看到中文提示。

## 五、风险分析评估

| 风险 | 等级 | 影响 | 当前缓解 | 后续建议 |
|---|---|---|---|---|
| PDF 扫描件无法抽取文本 | 中 | 简历解析失败 | 返回中文 `PARSE_FAILED` | 引入 OCR 或前端提示“仅支持可复制文本 PDF” |
| LLM API 不可用或超时 | 中 | 画像质量下降 | 正则兜底 | 增加调用耗时日志、重试、模型降级 |
| 正则解析准确率有限 | 中 | 技能/项目提取不完整 | 常见技能词表兜底 | 建立可配置技能词典与行业词库 |
| 测试内存库与生产 PostgreSQL 差异 | 中 | JSON/UUID 行为可能不同 | 覆盖业务契约 | CI 增加 PostgreSQL 测试矩阵 |
| 会话 token 为 MVP 简单 token | 高 | 生产安全不足 | 仅开发演示使用 | 使用 HMAC/JWT，绑定 TTL 和 sessionId |
| 上传文件本地存储 | 中 | 多实例部署不可共享 | 本地兜底 | 接入 TOS 私有对象存储 |
| 点餐/会议可直接创建会话 | 低 | 用户误以为完整功能已上线 | 前端显示 P1/P2 占位 | 后续增加后端功能开关拦截策略 |
| 前端测试依赖独立 package | 低 | 依赖版本可能与主前端漂移 | 锁定 React 版本 | CI 使用 workspace 或 pnpm workspace 管理 |

## 六、代码质量评估

### 优点

- 模块边界清晰，`routers`、`services`、`models` 分工明确。
- API 字段统一 camelCase，符合接口契约。
- 简历/JD 解析具备 LLM 和正则双路径，满足 MVP 72h 可落地要求。
- 错误响应统一，便于前端展示与测试断言。
- 测试按模块拆分，颗粒度适中，既覆盖单元逻辑，也覆盖端到端 API 链路。

### 待改进点

- `llm_service.py` 中技能词表是静态常量，后续应改为配置化。
- `resume_service.py` 当前 PDF 解析只支持文本层，不支持 OCR。
- `interviews.py` 当前 sessionToken 是简单字符串，生产前必须替换。
- `report` 与 `events` 当前是占位，后续需补充真实业务生成逻辑。
- 前端实时会话页目前只展示会话信息，Step 3 需补 WebSocket 收发与断线恢复。

## 七、优化建议

1. 后端 CI 增加三类测试命令：
   - `pytest backend -m p0`
   - `pytest backend -m contract`
   - `pytest backend -m integration`
2. 前端 CI 增加 `npm test` 与 `npm run build` 双检查，避免测试通过但 Next 构建失败。
3. 为 `POST /api/resumes` 增加文件大小、MIME、恶意文件名测试。
4. 为 `POST /api/interviews` 增加 token 生成、TTL、非法 UUID、跨用户访问测试。
5. 为 LLM 解析增加 mock LLM 成功路径测试，覆盖结构化 JSON 清洗逻辑。
6. 将 `tests/docs/代码测试文档.md` 作为 PR 描述的测试依据，保持每个 PR 更新。

## 九、验收结论

当前测试设计已覆盖 Step 2 的核心业务链路、接口契约、异常路径和前端关键组件交互。只要后端 `pytest backend -v` 与前端 `npm test` 全部通过，即可认为本阶段代码具备进入 PR 评审的基础质量。
