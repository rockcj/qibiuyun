# Agile Feature Delivery

## 使用场景

当用户要求开发新功能、拆分功能、安排 PR、准备提交、补测试或制定实现顺序时，使用本技能。目标是保证 OfferGPT 始终以可运行主流程为中心，按 P0、P1、P2 增量交付。

## 优先级判断

按必要程度和重要性排序：

| 优先级 | 定义 | 判断标准 |
|---|---|---|
| P0 | 主流程必需 | 不做就无法完成演示闭环 |
| P1 | 核心体验增强 | 不做仍可跑通，但体验或说服力下降 |
| P2 | 加分项 | 不做不影响主流程和比赛演示 |

默认先交付 P0。只有 P0 可运行、可测试、可提交后，才能进入 P1；只有 P1 稳定后，才进入 P2。

## 开发工作流

1. 明确目标：用一句话写清本次功能解决什么问题。
2. 标注优先级：判断功能属于 P0、P1 还是 P2，并说明原因。
3. 拆分模块：把功能拆成多个可独立测试的小模块。
4. 定义验收：每个模块写清输入、输出、页面或接口表现。
5. 先列测试：实现前明确自动测试、接口测试或手动验证方式。
6. 小步实现：每次只改当前模块相关文件。
7. 运行测试：相关测试通过后才允许准备提交。
8. 准备 PR：PR 只包含一个功能，并写清功能描述、实现思路和测试方式。

## 测试门禁

提交前必须满足：

- 当前功能的测试已通过。
- 已验证前序 P0/P1 功能仍能使用。
- 如果缺少自动测试，必须给出手动验证步骤和结果。
- 测试失败时停止提交，先修复问题再重新测试。
- 不允许为了提交而跳过测试、删除测试或弱化断言。

## PR 拆分规则

- 一个 PR 只交付一个细分功能模块。
- 不把 P0、P1、P2 混在同一个 PR。
- 不把功能开发和大规模重构混在同一个 PR。
- 后续功能可以依赖前序功能，但不能破坏前序功能可用性。
- 如果必须引入临时 Mock，要在 PR 描述中说明替换计划。

## Agent 执行要求

- 回答和文档使用中文。
- 变量名、函数名和接口字段使用 camelCase。
- 代码注释使用中文，说明关键业务逻辑。
- 开发前先给出模块拆分和测试计划。
- 开发后必须运行相关测试或说明无法运行的原因。
- 未通过测试时，不要提交，不要创建 PR，不要声称功能完成。
