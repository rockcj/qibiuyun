# PR 与持续交付规范

## 基本原则

- 所有回答、说明、提交信息和 PR 描述优先使用中文。
- 代码、功能和文档必须确保自主完成，不抄袭、不直接搬运他人作品。
- 如引用第三方库、框架、模板或复用过往代码，必须在 README 或 PR 描述中明确说明来源、用途和原创部分。
- 所有 commit 时间戳必须位于所选批次的开始时间与截止时间之间。

## PR 规则

- 必须基于 PR 添加或修改功能，不允许直接把大批量代码一次性导入主分支。
- 每个 PR 只做一件事，只实现或修改一个独立功能。
- 大功能必须拆分为多个小 PR，按可运行、可验证的阶段逐步提交。
- PR 合并后，主分支必须保持可运行状态，评委在任意时间查看都应能复现演示效果。

## PR 描述要求

每个 PR 必须包含以下内容：

- 标题：一句话说明本 PR 新增或修改了什么。
- 功能描述：说明该功能的作用、入口和使用方式。
- 实现思路：简要说明技术选型、核心逻辑或关键代码结构。
- 测试方式：说明如何运行、如何验证功能正常。

## 禁止事项

- 禁止 PR 描述空白。
- 禁止 PR 描述与实际代码变更严重不符。
- 禁止最后一天一次性导入全部代码。
- 禁止未说明依赖来源、第三方框架或复用代码来源。
- 禁止一个 PR 同时混入多个无关功能、格式化、重构或文档变更。

## 代码风格

- 使用驼峰命名法。
- 方法名和变量名必须具备描述性，避免使用无意义缩写。
- 代码注释使用中文。
- 关键逻辑、复杂判断、异步流程和业务规则必须添加简洁中文注释。
- 不要为显而易见的代码添加重复注释。

## 提交前检查

提交或创建 PR 前必须确认：

- 项目可以正常安装依赖并运行。
- 本 PR 的功能可以单独验证。
- README 已列明依赖、运行方式和原创功能说明。
- PR 描述完整，且和代码变更一致。
- 没有提交密钥、账号、`.env`、构建缓存或无关临时文件。
